# Roadmap: Nemovia

## Overview

Nemovia delivers a sagre aggregator for the Veneto region in 6 phases. We start by laying the foundation (project, database, deploy pipeline, design system), then build the data pipeline that scrapes from 5+ sources, enrich that data with LLM tagging and geocoding, build the discovery UI on top of real data, add interactive map and detail pages, and finish with SEO and visual polish. Each phase produces a verifiable, working increment.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

- [ ] **Phase 4: Discovery UI** - Homepage, sagra cards, search page with filters, quick filter chips

- [ ] **Phase 6: SEO & Polish** - Dynamic metadata, sitemap, OG images, premium animations, loading states

### Phase 1: Foundation & Design System
**Goal**: A deployed Next.js app on Vercel with Supabase PostGIS database, mobile-first layout shell, and brand design system ready for feature development
**Depends on**: Nothing (first phase)
**Requirements**: UI-01, UI-02, UI-03
**Success Criteria** (what must be TRUE):
  1. Next.js app is deployed on Vercel and accessible via public URL
  2. Supabase database exists with sagre table including PostGIS geography column
  3. Mobile layout shell renders with BottomNav (Home/Cerca/Mappa tabs) on phone viewport
  4. Brand colors (amber-600 primary, olive/green-700 accent, stone-50 bg) are applied across the shell
  5. pg_cron extension is enabled in Supabase

### Phase 2: Scraping Pipeline
**Goal**: Automated data collection that scrapes sagre from 5+ Veneto sources twice daily, deduplicates entries, and expires past events
**Depends on**: Phase 1
**Requirements**: PIPE-01, PIPE-02, PIPE-04, PIPE-05, PIPE-06
**Success Criteria** (what must be TRUE):
  1. Running the scraper populates the database with sagre from at least 5 distinct sources (SagreItaliane, EventieSagre, SoloSagre, TuttoFesta, Sagritaly)
  2. Adding a new source requires only a database config entry (CSS selectors, URL pattern), not code changes
  3. Duplicate sagre from different sources are merged into a single record (same name + city + overlapping dates)
  4. Past events are automatically marked inactive and excluded from queries
  5. Scraping runs automatically on schedule via pg_cron + Supabase Edge Functions

### Phase 3: Data Enrichment
**Goal**: Every scraped sagra has GPS coordinates, food/feature tags, and a compelling short description generated by LLM
**Depends on**: Phase 2
**Requirements**: PIPE-03, PIPE-07, PIPE-08, PIPE-09
**Success Criteria** (what must be TRUE):
  1. Every sagra with a city name has latitude/longitude coordinates from Nominatim geocoding
  2. Every sagra has food_tags (e.g., Pesce, Carne, Vino) and feature_tags assigned by Gemini 2.5 Flash
  3. Every sagra has an enriched description (max 250 characters) that reads as engaging and informative
  4. LLM calls batch 5-10 sagre per prompt to stay within the 250 RPD free tier limit
