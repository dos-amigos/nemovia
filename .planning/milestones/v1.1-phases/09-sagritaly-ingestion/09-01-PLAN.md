<research_findings>
## Critical Discovery: sagritaly.com is NOT JS-rendered

The roadmap assumed sagritaly requires "an approach that handles JS-rendered content." This is WRONG.

**sagritaly.com is a WordPress/WooCommerce site** that serves full server-rendered HTML (265KB per page). Cheerio can parse it directly. The domain is sagritaly.com (not sagritaly.it which is unreachable).

### Active Veneto Events URL

**Best URL:** `https://sagritaly.com/categoria/eventi-e-sagre/?filter_pa_regioni-sagre=veneto`

This returns ~10 active (upcoming/current) Veneto events in a single page. Pagination does not work with the filter param (returns 404), so max_pages=1.

### HTML Structure (active events page)

Each event is an `<article>` element. Key selectors:

- **Item container:** `article.w-grid-item` (class includes WP post metadata like `pa_province-sagre-venezia pa_regioni-sagre-veneto`)
- **Title:** `h5.post_title a` text (note: h5 not h3 on active events page)
- **Start date:** `div.data_inizio span.w-post-elm-value` -- format "DD/MM/YYYY"
- **End date:** `div.data_fine span.w-post-elm-value` -- format "DD/MM/YYYY"
- **City:** `div.luogo_evento span.w-post-elm-value` -- e.g., "Forte Marghera", "Valdobbiadene"
- **Province:** taxonomy span like `span.term-venezia` text inside `div.post_taxonomy` -- e.g., "Venezia", "Padova"
- **Image:** `img.wp-post-image` src attribute
- **Detail URL:** `h5.post_title a` href -- links to sagritaly.com detail page
- **External URL:** `a[target="_blank"]` at article top (optional, links to event's own website)
- **Description:** `div.post_content` text (truncated excerpt)

    <a target="_blank" href="https://www.facebook.com/gattorosso.mestre" aria-label="Festa del Pesce di Forte Marghera"></a>

### Date Format

Dates are DD/MM/YYYY format (e.g., "06/02/2026"), which the existing `parseItalianDateRange()` already handles via its slash-match Pattern 1.

### Passati (Past Events) Page

The `/regioni-sagre/veneto/` URL also has server-rendered HTML with a slightly different layout (h3 titles, different CSS class numbering) but the same data fields. These are past events. We use the active events URL instead.

```typescript
// --- sagritaly-specific extraction ---
// WordPress/WooCommerce site with structured custom fields for dates and location
if (source.name === "sagritaly") {
  const $el = $(el);

  // Title: h5 on active events page, h3 on passati page
  const title = $el.find("h5.post_title a, h3.post_title a").first().text().trim();

  // Dates: custom fields with class containing data_inizio / data_fine
  // Format is DD/MM/YYYY which parseItalianDateRange() handles via Pattern 1
  const startDateRaw = $el.find("div[class*='data_inizio'] span.w-post-elm-value").first().text().trim();
  const endDateRaw = $el.find("div[class*='data_fine'] span.w-post-elm-value").first().text().trim();
  const dateText = endDateRaw ? `${startDateRaw} al ${endDateRaw}` : startDateRaw;

  // City: custom field with class containing luogo_evento
  const city = $el.find("div[class*='luogo_evento'] span.w-post-elm-value").first().text().trim();

  // URL: from the title link (sagritaly.com detail page)
  let url = $el.find("h5.post_title a, h3.post_title a").first().attr("href") ?? null;
  if (url && !url.startsWith("http")) {
    try { url = new URL(url, source.base_url).href; } catch { /* */ }
  }

  // Image: WordPress post thumbnail
  let image = $el.find("img.wp-post-image").first().attr("src") ?? null;
  if (image && !image.startsWith("http")) {
    try { image = new URL(image, source.base_url).href; } catch { /* */ }
  }

  return { title, dateText, city, price: null, url, image };
}
```
